# 이정준 | 프론트엔드 개발자

> 사용자의 니즈와 페인포인트를 파악하여 고객 경험 혁신을 위해 노력하는 프론트엔드 개발자입니다.

<br />

## 👋 소개

프론트엔드 개발자로 근무하면서 부동산 종합 플랫폼을 개발하고 있으며, 복잡한 비즈니스 요구사항을 기술적으로 해결하고 유지보수성 높은 코드를 작성하기 위해 노력하고 있습니다.

본 레포지토리는 실무에서 직면했던 기술적 문제와 해결 과정을 정리한 내용입니다.

<br />

## 📁 주요 프로젝트 경험

#### 1. [채권담보 분석(얼마니) 서비스 구현 - 동적 UI 구조화](./ulmaniai/README.md/)

> 다양한 매물 유형 × 점유 상태 조합에 따른 동적 UI를 유틸 함수와 커스텀 훅으로 구조화한 경험

**문제 상황**
[채권담보 이미지]

- 매물 유형(단독주택, 다가구, 상가주택, 아파트 등)과 점유 상태(거주중/임대중/거주+임대중)의 조합에 따라 각기 다른 UI 필요
- 총 가구 수, 임차인 수 변경 시 테이블 행 자동 생성/삭제 필요
- 고객사 요구에 따라 수시로 변경되는 상황

**해결 방법**

- `autoSetUsageStatus` 유틸 함수로 점유 상태별 행 생성 로직 분리
- `useOccupancySection` 커스텀 훅으로 상태 관리 캡슐화
- `COMPONENT_MAP`으로 매물 유형별 컴포넌트 자동 매핑

**결과**

- 매물 분류 키 하나로 어떤 UI 조합이든 자동 렌더링
- 기획 변경 시 유틸 함수만 수정하면 되어 유지보수 용이
- 새로운 매물 유형 추가 시 한 줄만 추가하면 됨

→ [상세 코드 보기](./ulmaniai/README.md/)

### 얼마니 매물관리 (부동산 매물관리 서비스)

### 2. [매물관리 서비스 - 상태관리 리팩토링](./expert-ulmaniai/README.md)

> 기능 추가 요청을 처리하는 과정에서 기존 코드의 구조적 문제를 발견하고 개선한 경험

**문제 상황**
[매물관리 페이지네이션 이미지]

- react-hook-form + useWatch로 단순 상태를 관리하면서 불필요한 복잡도 증가
- 데이터 흐름이 form context에 숨어있어 추적 어려움
- 필터 변경 시 page=1 리셋 로직이 라우터 이동 시에도 동작하는 버그

**해결 방법**

- react-hook-form → useState + props 전환 (명확한 단방향 데이터 흐름)
- URL을 Single Source of Truth로 설정
- 초기화 플래그 패턴으로 첫 마운트와 필터 변경 구분

**결과**

- 필터 변경 시 page=1 리셋, 라우터 이동 시 페이지 유지 정상 동작
- useWatch 제거로 불필요한 리렌더링 감소
- Props 인터페이스로 명확한 컴포넌트 계약, 재사용성 향상

→ [상세 설명 보기](./expert-ulmaniai/README.md)

## 💡 개발 철학

1. **단순히 동작하는 코드가 아닌, 유지보수하기 좋은 코드**를 작성하려고 노력합니다.
2. **문제를 발견하면 근본 원인을 파악**하고, 필요하다면 리팩토링을 먼저 진행합니다.
3. **변경에 유연한 구조**를 고민하여 비즈니스 요구사항 변화에 빠르게 대응합니다.
